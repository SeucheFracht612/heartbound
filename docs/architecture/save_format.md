# Save Format Architecture

Save data must be versioned from day one.

Permanent save identity must not use raw pointers, ECS indices, physics ids, or array
positions.

Identity concepts:

```text
SaveId         = permanent object identity
NetId          = multiplayer session identity
RuntimeHandle  = current process identity
PrototypeId    = stable string id from mods/content
```

The current codebase provides identity primitives, save metadata validation, save id
reservation, deterministic per-mod prototype fingerprints, an ordered save migration
registry, a metadata text codec, a typed snapshot text codec, a typed binary snapshot
codec, a file-backed save database for full snapshots plus chunk-delta payloads, and a file-backed
save-slot catalog for safe per-slot database roots and validated per-slot display metadata.
The file save database has both raw snapshot reads for storage/migration code and a validated read
path for load code that has an active prototype registry.

The current save snapshot model already keeps typed sections for chunk edits, build
pieces, entities, inventories, cargo, workpieces, assemblies, processes, and mod state.
The validator checks stable ids, prototype kinds, and cross-section references so these
sections do not collapse into one generic saved-object blob.

The text snapshot codec is an early persistence format for tests, tools, and golden
fixtures. The binary snapshot codec is the first compact full-snapshot backend. Both
write one explicit record kind per saved representation and round-trip all current
`SaveSnapshot` sections. The first file-backed database stores the binary snapshot and
chunk deltas separately; future database formats should preserve the same typed-section
boundary.
The save-slot catalog owns display metadata and timestamp policy; callers that commit through the
catalog update `last_saved_at_ms` after the underlying slot database accepts the snapshot.

Persistent entity records save stable identity and transform state, while runtime handles and
session net ids are regenerated by the world snapshot bridge on load.
The prototype-aware world snapshot import path validates saved prototype ids and expected kinds
against the active registry before materializing runtime world records.

New saves build `SaveMetadata::enabled_mods` from the active content validation fingerprints
through `content::save_metadata_from_content_report`. Each saved mod record stores the mod id, mod
version, and normalized prototype hash so later load/migration code can detect content drift without
depending on real disk paths.

`SaveMetadata::validate` rejects duplicate enabled mod ids, empty migration history entries, and
duplicate migration history entries. Migration history is an ordered audit trail, not a set of
arbitrary repeatable notes.

`SaveCompatibilityChecker` compares saved mod records against the active mod prototype
fingerprints. Missing saved mods, prototype-hash changes, and extra active mods are compatibility
errors until saves persist the runtime voxel-palette mapping needed to prove that compact numeric
cell types cannot be reinterpreted. Version differences remain migration-policy diagnostics.
